#1 Trigger.dev：长任务 AI Agent 执行器

GitHub

Trigger.dev 的价值不是“任务调度平台”，而是把容易超时、失败、需要重试和人工确认的 AI Agent 工作流，变成可观测、可恢复、可排队的后台执行器。

解决的问题：Codex / Claude Code / 自写脚本可以生成流程，但真正跑日报、抓取、转码、分析时经常遇到超时、失败不可追踪、人工审核节点缺失的问题。

主流方案对比：Cron + Vercel Functions 或普通 serverless 适合短任务；Trigger.dev 更适合长任务、重试、队列、checkpoint、人审和运行日志，适合作为 AI Opportunity Radar 的稳定执行层。

你现在是我的 AI Agent 后台执行架构师。 目标：基于 Trigger.dev，设计一个“AI Opportunity Radar 长任务执行器”，用来每天稳定执行信息抓取、候选筛选、AI 摘要、JSON 写入和 Dashboard 更新。 请不要介绍 Trigger.dev 是什么，直接输出可开发方案。 请按下面结构输出： 1. 这个执行器要解决的具体稳定性问题 2. 哪些步骤应该作为 Trigger.dev task，哪些仍然保留在普通脚本里 3. 设计 1 条 daily-radar-run task：输入参数、执行步骤、输出结果 4. 重试规则：GitHub Search 失败、网页抓取失败、AI 摘要失败、GitHub 写入失败分别怎么处理 5. 人工审核节点：在哪一步 pause 等待人工确认最有价值 6. 日志和可观测字段：每次运行必须记录哪些字段 7. 最小实现目录结构和伪代码 8. 今天 30 分钟内如何先用 mock 数据验证闭环 限制： - 不要做大而全平台 - 不要改现有 Dashboard UI - 只围绕“让日报准时、稳定、可追踪”设计 - 输出要能直接交给 Codex 开发。
